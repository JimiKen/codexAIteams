# 项目经理文档（PM）

## 1. 项目概况

- 项目名称：Codex AI Teams（PySide6 桌面端）
- 当前路径：`D:\codexAIteams\aitesms`
- 当前版本基线：`v1.14`（见 `需求文档.md` 与 `功能清单.md`）
- 当前范围：团队/配置/日志/文件/终端 5 页面 + 多 Agent 并发对话

## 2. 本阶段目标

- 保持 5 页面主流程稳定可用。
- 完成“基础版”能力补齐：终端实时流式、日志高级筛选、文件预览搜索、配置校验。
- 建立每轮迭代的文档同步机制（需求、功能清单、角色文档）。

## 3. 任务拆解（WBS）

1. 需求与范围管理
- 对齐 `需求文档.md`、`功能清单.md` 与代码实现差异。
- 每轮变更标注 新增/修改/删除。

2. 版本计划与里程碑
- 里程碑 M1：终端页实时流式输出（stdout/stderr 增量刷新）。
- 里程碑 M2：日志页新增 Agent/时间范围筛选。
- 里程碑 M3：文件页新增文件预览与搜索。
- 里程碑 M4：配置页新增字段校验与导入导出模板。

3. 过程治理
- 建立每周节奏：需求冻结 -> 开发 -> 回归 -> 发布。
- 统一验收清单与风险清单。

4. 发布与复盘
- 发布前确认关键路径（团队对话、配置保存、日志导出）。
- 发布后记录问题、原因、改进动作。

## 4. 排期建议（2 周迭代）

1. 第 1-2 天
- 需求冻结、任务拆解、风险评审、确认验收标准。

2. 第 3-7 天
- FE/BE 实施 M1 + M2，QA 同步设计用例。

3. 第 8-10 天
- FE/BE 实施 M3 + M4，联调与缺陷修复。

4. 第 11-12 天
- 全量回归、性能与稳定性检查。

5. 第 13-14 天
- 发布、文档归档、迭代复盘。

## 5. 关键风险与应对

1. 外部 Codex CLI 不稳定或超时
- 应对：增加超时分级、失败重试策略与错误码分类统计。

2. 多 Agent 并发导致 UI 卡顿
- 应对：严格线程隔离，流式日志限长与节流。

3. 配置错误导致运行异常
- 应对：配置保存前校验（必填项、数值范围、格式）。

4. 需求频繁变更导致返工
- 应对：迭代中期后仅接收高优先级变更，其他进 backlog。

## 6. 验收口径

- 5 页面均可打开并完成核心动作。
- 群发/单发对话可触发多 Agent 并发执行。
- `config/teams.yaml` 可稳定读写。
- 日志可筛选且可导出 CSV。
- 文档与代码版本一致，无“功能已实现但文档缺失”。

## 7. 协作接口

- 对前端：提供优先级、交互验收标准、发布时间点。
- 对后端：提供接口行为约束、错误处理标准。
- 对测试：提供测试范围、回归重点、发布门禁标准。
